## What this project is

A web-based Reed-Kellogg sentence diagramming game. Originally built as a
Dev.to Weekend Challenge submission вЂ” Flask backend, PostgreSQL database,
vanilla JS + SVG frontend, Docker containerized. The goal of this session
was to prepare it for a Slingcode port: a self-contained single HTML file
with no server, no Docker, distributable as a file.

### 6. Architecture decisions made

**Slingcode port pipeline:**
```
Postgres (authoring) в†’ pg_export.py в†’ sentences.yaml в†’ build_slingcode.py в†’ sr_barbara.html
```

**JS module split planned** (game.js в†’ 6 modules):
- `config.js` вЂ” colors, labels, Sr. Barbara dialogue, layout constants
- `layout.js` вЂ” calculateLayout(), estimateWidth()
- `renderer.js` вЂ” SVG drawing functions
- `ui.js` вЂ” DOM manipulation, legend, feedback, popup
- `data.js` вЂ” Flask version: fetch('/api/sentence')
- `data.slingcode.js` вЂ” Slingcode version: local SENTENCES lookup
- `main.js` вЂ” event wiring, app initialization

**Update system design:**
- GitHub Releases API (unauthenticated, public repo)
- Badge/dot indicator in a settings/profile panel вЂ” not intrusive
- Check only on user action, never automatic
- Game works fully offline, update check fails silently if offline
- GitHub Actions pipeline: push в†’ srb-build в†’ GitHub Release в†’ asset upload
- In-game checker fetches `/repos/carolynboyle/sr-barbara/releases/latest`
- Players never need GitHub account or token

**Config externalization:**
- `sr_barbara.yaml` at repo root вЂ” consumed by ALL scripts in the pipeline
  as the first thing each script does. No script hardcodes paths or makes
  assumptions about where things live.
  - `pg_export.py` reads it for: `paths.sentences_yaml` (output path),
    `database.env_file` (where to find credentials)
  - `build_slingcode.py` reads it for: `paths.sentences_yaml` (input),
    `paths.build_output` (where to write sr_barbara.html),
    `game.default_difficulty` (default selected difficulty in built HTML)
  - Any future script follows the same pattern вЂ” load config first.
- `pyproject.toml` вЂ” single source of truth for version (read by fletcher,
  setupkit, and the GitHub Actions release pipeline)
- `data/sentences.yaml` вЂ” single source of truth for sentence data
- viewkit pattern applied to `app.py` queries (queries.yaml)
- JS config object from `game_config.yaml` via build script
